## Context

Deeply is an AI product for couples: each partner privately shares their perspective, and the system produces **one shared synthesis** that both people recognize as fair. I joined as a **founding engineer** on a small team and own core product surfaces end-to-end — from prototype through the shipped artifact — across the production app and a separate strategy/prototyping repo.

## Problem

A two-player AI experience is fundamentally harder than a single-user chatbot:

- The output has to feel **fair to two people at once**, not just coherent to one.
- LLM behavior is **non-deterministic**, so "it worked when I tried it" is not a guarantee.
- Every change to a prompt or tool can silently regress quality for real couples.

The core problem was building this as a **dependable production system**, not a demo.

## Architecture

The system is a production LLM agent built on the **Anthropic Claude SDK**, with a few moving parts I own:

- **Conversation engine** — orchestrates the two-player flow and turn-taking.
- **Agent tools** — the actions the model can take during a session.
- **Context assembly** — what each partner's private input contributes to the shared synthesis.
- **Artifact generation** — producing the final shared output both partners see.

Around the agent sits the platform: **Next.js 14 / React** app, **Supabase (Postgres)** for data, partner **invite / deep-link** flows, **Slack + in-app notifications**, and internal **admin tooling** for inspecting live AI conversations (system prompts, tools, artifacts).

```text
  Partner A ──┐                 ┌── Partner B
  (private)   │                 │   (private)
              └────────┬────────┘
                       ▼
              [ Conversation Engine ]   two-player turn-taking
                       │
                       ▼
              [ Agent Tools ]           actions the Claude SDK agent can take
                       │
                       ▼
              [ Context Assembly ]      private inputs → one shared context
                       │
                       ▼
              [ Artifact Generation ]   the synthesis both partners see
                       │
                       ▼
              [ Shared Artifact ]

  Around the agent: Langfuse (prompts + traces) · Sentry (errors) · PostHog (analytics)
```

## Implementation

A few things I'm responsible for that make the AI layer trustworthy:

- **Eval-driven prompt engineering.** Prompts and tracing live in **Langfuse**; I authored a custom **eval harness** (scenarios + assertions) that catches regressions before they ship.
- **Per-PR ephemeral databases.** I designed a **Supabase branch-database CI/CD system** so every pull request gets an isolated, seeded Postgres database, with automatic apply-on-merge.
- **End-to-end observability.** **Sentry** for errors, **PostHog** for analytics, a **cron watchdog**, and a public **status page**.
- **Admin tooling.** Internal views to inspect live AI conversations — the exact system prompts, tools, and artifacts — so we can debug behavior, not guess at it.

In a separate strategy repo, I also built design-system-driven prototypes and a **GitHub Actions → GitHub Pages** pipeline that auto-converts Markdown/HTML into a branded, searchable static site, giving non-technical stakeholders self-serve access to live mockups on every merge.

## Key Tradeoffs

- **Evals + tracing cost time up front** but turn "I think this is better" into "the assertions pass." For an AI product, that's the only way to move fast safely.
- **Ephemeral per-PR databases** add CI complexity and cost, but remove an entire class of "it worked on my branch" bugs and make AI changes genuinely reviewable.
- **Admin/inspection tooling** isn't user-facing, but it's where most debugging time is saved.

## Results

- Built and own core surfaces of a **production LLM agent system** powering a two-player couples experience.
- Every pull request runs against an **isolated, seeded database**, with apply-on-merge.
- Production is instrumented end-to-end — errors, analytics, a watchdog, and a public status page.

## Lessons

- **Evals are infrastructure.** Treating them like tests is what lets a small team ship AI changes without fear.
- **Tracing turns debugging into observation** — with LLMs, you cannot fix what you cannot see.
- **Owning a surface means owning its reliability** — the invite flow, the notifications, the database lifecycle, and the observability around it, not just the happy path.
