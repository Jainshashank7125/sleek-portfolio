## Context

A **healthcare provider credentialing** platform must continuously check that providers meet compliance requirements — state licenses, DEA/CDS certifications, board certifications, malpractice insurance, sanctions, and NPDB reports. I designed and maintain the **Drools rules engine** that encodes and enforces this logic.

## Problem

Credentialing rules are **high-volume, high-stakes, and payer-specific**:

- There are **hundreds** of distinct compliance rules that must stay correct and auditable.
- **15+ health plan organizations** each have their own variations of the logic.
- Rules change often, and changing them **must not require redeploying** the whole platform.

The challenge was encoding all of this on **one shared engine** without organizations interfering with each other.

## Architecture

- A library of **300+ Drools DRL rules** enforcing credentialing compliance across all the certification and sanction categories.
- **Multi-tenant rule isolation** — each of the 15+ organizations runs its own payer-specific logic on the shared engine, with no cross-org leakage.
- A **`rulesConfiguration` map** that drives **feature-flag-based rule execution**, enabling per-organization runtime toggling and A/B rollouts with **zero redeployment**.

## Implementation

- **Real-time monitoring flag rules** detect expired board certifications, duplicate/expired state licenses, NPPES name mismatches, missing DEA data, and sanction findings — powering automated provider compliance alerts.
- A **shell-based test automation framework** (`run-tests.sh`) with **JSON payload fixtures** and **upsert-on-conflict expected-result matching** enables fast local rule validation and regression coverage across hundreds of scenarios.

## Key Tradeoffs

- **A rules engine vs. hand-coded conditionals:** Drools keeps volatile compliance logic out of application code and makes it auditable — at the cost of a learning curve and engine overhead.
- **Feature-flagged rules vs. branching code:** toggling rules via configuration enables zero-downtime rollouts, but requires disciplined flag hygiene.
- **Shared engine vs. per-tenant engines:** one engine is far cheaper to operate, but isolation has to be designed in carefully so one org's rules can never affect another's.

## Results

- **300+ DRL rules** maintained with regression coverage across hundreds of scenarios.
- **15+ organizations** served with isolated, payer-specific logic on a shared engine.
- Rule changes and rollouts ship with **zero redeployment** via runtime toggles.

## Lessons

- **A rules engine is the right tool** when business logic is large, volatile, and must be auditable.
- **Feature flags turn risky rule changes into config changes** — reviewable and reversible.
- **Test fixtures with expected-result matching** are what make a 300-rule system safe to evolve.
